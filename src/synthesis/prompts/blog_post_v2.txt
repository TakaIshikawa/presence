- Deeply practical and utility-focused
- Direct, reflective, shares both wins and struggles

Given: A week's worth of prompts and commits.

Interpret the source material this way:
- Prompts are the strongest signal: they show the founder's hypothesis, product concern, and evaluation target.
- Commits are supporting evidence: they show the tools, methods, workflows, constraints, and instrumentation introduced to test the hypothesis.
- The post should not summarize engineering output. It should explain what was learned about improving AI agent products.
- Write as the founder/operator explaining the experiment and its implications.

Write a blog post that:
1. Has a clear narrative arc: hypothesis -> intervention -> measurement -> result -> uncertainty
2. Names the intervention as a tool, method, technique, workflow, constraint, or feedback loop
3. Explains why that intervention was introduced
4. Explains how the outcome was measured or observed
5. Provides transferable principles other AI product builders can apply
6. 800-1200 words
7. Has 3-4 section headers (## format)

IMPORTANT - DO NOT:
- Describe specific features built or bugs fixed
- Mention error codes, API names, or library-specific details
- Focus on what your project does
- Write technical documentation
- Write as if the coding agent is reporting its own actions
- Treat commits as the narrative; they are evidence only
- Use vague platitudes
- Invent metrics, percentages, or statistics not in the source material
- Fabricate outcomes ("reduced by X%", "N times faster")

INSTEAD focus on:
- The product hypothesis: what agent behavior or user outcome needed to improve
- The intervention: what tool/method/technique changed the system
- The measurement: benchmark, acceptance test, user signal, latency, completion rate, quality score, failure rate, review burden, or another source-backed signal
- The result: what changed, how large the signal was if measured, and how reliable the number is
- The tradeoff: what the intervention made harder, slower, more expensive, or more constrained
- The reusable principle for AI agent product builders
- Use source wording for interventions and signals; do not invent polished labels the evidence does not contain
- Keep causal claims narrow: do not imply production/user improvements unless the source includes that signal
- Ground every claim in the source material — a truthful observation beats an impressive-sounding lie
- Do not invent umbrella theories, named principles, or polished hypotheses that are not explicit in the prompts
- If the source only shows that a tool/check/export/analyzer was introduced, describe the intended measurement loop and the uncertainty; do not claim a proven product outcome
- Choose the least-repeated source-backed angle available from THIS WEEK'S ACTIVITY, even if that means a narrower post
- Use numbers only when present in the source material; if no metric exists, be explicit that the evidence is qualitative

The title should follow the pattern: "[What we learned about improving AI agent products]"

Example good titles:
- "Why Agent Reliability Improved When We Added a Refusal Path"
- "The Evaluation Harness Changed the Product More Than the Prompt"
- "A Better Agent Starts With a Better Measurement Loop"

Example bad titles (too project-specific):
- "Building My Auth System with Claude"
- "How I Fixed Database Performance This Week"
